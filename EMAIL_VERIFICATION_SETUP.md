# Подтверждение Email — настройка и проверка

Верификация email **полностью реализована в коде** (генерация 6-значного кода,
хранение bcrypt-хэша с TTL, одноразовость, защита от перебора, ввод кода на сайте,
блокировка функций до подтверждения на UI + API + middleware).

Единственное, что зависит от внешней настройки, — **реальная доставка письма**.
Без SMTP всё работает в **dev-режиме** (код возвращается фронтенду и пишется в лог
backend), поэтому флоу можно протестировать прямо сейчас. Для продакшена подключите SMTP.

---

## Как это работает

1. Пользователь регистрируется → создаётся аккаунт с `emailVerified = null`.
2. Backend генерирует 6-значный код, сохраняет **только его bcrypt-хэш**
   (`verification_codes`) с `expiresAt = now + 15 мин`, и отправляет письмо.
3. Пользователь попадает на `/verify-email` и вводит код.
4. `POST /api/auth/verify-email` сверяет код (bcrypt), при успехе ставит
   `emailVerified = now`, гасит код (`consumedAt`) и выдаёт новый JWT с `verified: true`.
5. До подтверждения middleware отдаёт `403 EMAIL_NOT_VERIFIED` на защищённые
   функции (практика, экзамены, игры, покупка Premium, избранное и т.д.).
   Разрешены только `/api/users/me`, `/api/auth/verify-email`, `/api/auth/resend-code`.

**Параметры** (в `lib/verification.ts`): TTL 15 мин, до 5 попыток ввода,
повторная отправка не чаще 1 раза в 60 с.

---

## Dev-режим (без SMTP) — работает из коробки

Если переменная `SMTP_USER` не задана:
- письмо не отправляется;
- код пишется в консоль backend: `[verify] DEV verification code for <email>: 123456`;
- код также возвращается в ответе `register` / `resend-code` как поле `devCode`,
  и фронтенд показывает его на странице `/verify-email` в жёлтой плашке.

Этого достаточно, чтобы протестировать весь сценарий локально.

⚠️ В продакшене обязательно задайте SMTP — тогда `devCode` **не возвращается**
(поле становится `undefined`), и код приходит только на почту.

## Подключение реальной почты (продакшен)

### 1. Выберите SMTP-провайдер
Любой из:
- **Resend** (`smtp.resend.com`, порт 587) — просто, есть бесплатный тариф;
- **SendGrid** (`smtp.sendgrid.net`, 587);
- **Mailgun** (`smtp.mailgun.org`, 587);
- **Gmail** (`smtp.gmail.com`, 587) — только с «паролём приложения», для теста;
- корпоративный SMTP вашего домена.

### 2. Добавьте переменные окружения
Backend читает их в `ct-platform/backend/lib/email.ts`.

Локально — в `ct-platform/backend/.env`:
```
SMTP_HOST=smtp.your-provider.com
SMTP_PORT=587
SMTP_USER=ваш_логин_или_apikey
SMTP_PASS=ваш_пароль_или_apikey
EMAIL_FROM="CT-Platform <noreply@ваш-домен.by>"
```
На Render (backend) — те же переменные в **Environment** сервиса.

> Примечание: текущий транспорт использует `secure: false` (STARTTLS, порт 587).
> Если провайдер требует SSL на порту 465 — поставьте `secure: true` для порта 465
> в `lib/email.ts` (одна строка: `secure: parseInt(SMTP_PORT) === 465`).

### 3. Проверьте отправителя
Большинство провайдеров требуют верифицировать домен/адрес отправителя
(`EMAIL_FROM`), иначе письма уйдут в спам или будут отклонены. Настройте SPF/DKIM
по инструкции провайдера.

### 4. Перезапустите backend
Render задеплоит при пуше; локально — перезапустите `npx next dev`.

## Как проверить, что всё работает

**Dev (без SMTP):**
1. Зарегистрируйте нового пользователя.
2. На `/verify-email` появится плашка с кодом (или возьмите код из лога backend).
3. Введите код → доступ открывается.
4. Попробуйте до подтверждения открыть практику/игру → увидите экран «Подтвердите email».

**Прод (со SMTP):**
1. Зарегистрируйтесь реальным адресом → письмо с кодом приходит на почту.
2. Введите код на сайте → `emailVerified` проставляется, выдаётся новый токен.
3. Повторная отправка — кнопка «Отправить код снова» (кулдаун 60 с).
4. Неверный код 5 раз → требуется запросить новый.
5. Код старше 15 минут → «Срок действия истёк, запросите новый».

## Затронутые файлы
- `prisma/schema.prisma` — модель `VerificationCode`, поле `User.emailVerified` (уже было).
- `lib/verification.ts` — генерация/выдача кода (новый).
- `lib/email.ts` — шаблон `getVerificationEmail` (+ существующий `sendEmail`).
- `lib/auth.ts` — клейм `verified` в JWT.
- `app/api/auth/register/route.ts` — выдача кода при регистрации.
- `app/api/auth/verify-email/route.ts`, `app/api/auth/resend-code/route.ts` — новые.
- `app/api/auth/login/route.ts`, `app/api/users/me/route.ts` — отдают `emailVerified`.
- `middleware.ts` — блокировка функций при `verified === false`.
- Frontend: `app/src/pages/VerifyEmailPage.tsx`, gating в `useAppStore`/`AuthGateModal`.

## Backfill уже выполнен
Существующие аккаунты (admin, demo и т.д.) помечены подтверждёнными
(`prisma/backfill-email-verified.js`), чтобы введение проверки их не заблокировало.
